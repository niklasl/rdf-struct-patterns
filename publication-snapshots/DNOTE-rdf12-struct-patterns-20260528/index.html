Abstract

The purpose of this document is to define semantic extensions for interpreting the formal structures of propositions and literal values, as defined in RDF 1.2 Concepts and Abstract Data Model. In addition, examples of semantic interoperability patterns using OWL in conjunction with these extensions are provided. This assumes knowledge of the basic concepts of RDF, and familiarity with reifiers, as exemplified in the RDF 1.2 Primer. It furthermore relies on the semantic conditions defined in RDF 1.2 Semantics. The section about OWL-based Interoperability patterns assumes an understanding of OWL 2 Web Ontology Language RDF-Based Semantics (Second Edition).

1. Introduction

This document defines semantic conditions for entailing the properties inherent in the formal structures of:

  1. RDF 1.2 propositions, encoded as triple terms;
  2. RDF 1.2 directional language strings and regular language strings, encoded as rdf:dirLangString and rdf:langString literals, respectively.

The structure and meaning of these abstractions enables further processing, by using OWL or other semantic extensions. Based on this, a set of interoperability pattern examples are examined, including a definition of classic reification as RDF 1.2 reifiers.

3. Properties of Formal Structures

In RDF, resources are regularly denoted by IRIs, or existentially quantified by blank nodes, and further described through statements about them, rendering their definitions more precise. This is not necessary for propositions and literal values, since their meaning is completely encoded by their corresponding terms—ground triple terms and literals—in the abstract syntax.

Propositions are fixed, invariant formal abstractions within the class extension denoted by rdfs:Proposition, having three intrinsic constituent components: the subject, predicate and object of the proposition. This structure forms a precise conceptual identity, corresponding to the triple terms, which are 3-tuples of RDF terms, acting as a compound key denoting the proposition through its structure within the class extension.

Note

Formally, blank nodes denote some resource, thus a triple term with a blank node denotes some proposition; though functionally dependent on its constituents.

Literal values, while fundamentally different concepts (some defined outside of RDF), share this fixed, invariant characteristic. One difference is that it requires its datatype to be recognized (to be in D), and if it does, it is fixed in all interpretations recognizing it.

Therefore, while these kinds of resources are conceptually just resources in the domain of discourse, they have a particular standing due to their formally fixed definitions. The RDF abstract syntax reflects this by only allowing literals and triple terms as objects of other triples.

Still, these invariant resources logically have properties. They are simply intrinsically defined, as an interpretation of the structural constituents of the resource within the space of their type.

In the following sections we will see how these facts can be formally entailed, and encoded using basic statements. The semantic extensions defined are based on the semantic conditions of RDFS interpretations, defined in RDF 1.2 Semantics.

Note

The SPARQL 1.2 Query Language defines a number of functions on RDF terms. In principle (and noting that functions—including those on IRIs—operate on lexical forms), the unary functions defined there can correspond to predicates of the formal structures which these terms denote (in the logical sense, and expressed in RDF as statements). More precisely, if F is a SPARQL function and P is its associated predicate, for all RDF terms S on which F is defined, the statement S P F(S) must be true.

3.1 Formal Structure of Propositions

An rdfs:Proposition instance is a resource having exactly one subject resource, one predicate property and one object resource.

The proposition pattern: 

XXX YYY <<( AAA BBB CCC )>> .

can thus be described as: 

XXX YYY _:pp .
_:pp rdf:propositionFormSubject AAA .
_:pp rdf:propositionFormPredicate BBB .
_:pp rdf:propositionFormObject CCC .

Note that the following is already entailed through RDFS semantics: 

_:pp a rdfs:Proposition .

And this, provided the axioms below: 

_:pp a rdf:PropositionForm .

3.1.1 Extended Proposition Interpretation

An extended proposition interpretation is an RDFS interpretation I which also satisfies the following semantic condition, and all the triples in the subsequent table of axiomatic triples.

Proposition semantic condition.

If I(E) is in ICEXT(I(rdfs:Proposition)), and I(E) = IT(I(s) ,I(p), I(o)) where
s is the subject, p is the predicate and o is the object component of E,
then:

  • <I(E), I(s)> is in IEXT(I(rdf:propositionFormSubject)),
  • <I(E), I(p)> is in IEXT(I(rdf:propositionFormPredicate)), and
  • <I(E), I(o)> is in IEXT(I(rdf:propositionFormObject)).
Axiomatic triples of proposition forms. 
rdf:PropositionForm rdf:type rdfs:Class .

rdf:propositionFormSubject rdf:type rdf:Property .
rdf:propositionFormSubject rdfs:domain rdf:PropositionForm .
rdf:propositionFormSubject rdfs:range rdfs:Resource .

rdf:propositionFormPredicate rdf:type rdf:Property .
rdf:propositionFormPredicate rdfs:domain rdf:PropositionForm .
rdf:propositionFormPredicate rdfs:range rdf:Property .

rdf:propositionFormObject rdf:type rdf:Property .
rdf:propositionFormObject rdfs:domain rdf:PropositionForm .
rdf:propositionFormObject rdfs:range rdfs:Resource .

3.2 Formal Structures of Literal Values

Literals are also denoted by tuple structures, specifically of an IRI denoting their datatype, and a string representing a lexical form. They are fundamentally different from propositions in that their lexical form may encode values of any kind of complexity, depending on its datatype. The structure so encoded is defined by the formal meaning of this datatype, which defines its value space. The result of decoding the lexical form according to this datatype is the literal value, having the datatype as its rdf:type, and other properties according to the formal meaning of its lexical encoding.

Some value spaces are formally specified in D-interpretations of [RDF-SEMANTICS], and in rdf:PlainLiteral: A Datatype for RDF Plain Literals (Second Edition). The following subsections extend these definitions by entailing statements about the literal values.

3.2.1 Directional Language Strings

A directional language string is a typed resource combining one string value with one rdf:language and one rdf:direction. This form acts as a key, uniquely identifying the literal value.

The directional language string pattern: 

SSS PPP "..."@und--rtl .

can thus be described as: 

SSS PPP _:dls .
_:dls rdf:value "..." .
_:dls rdf:language "und"^^xsd:language .
_:dls rdf:direction "rtl"^^rdf:langDir .

Note that the following is entailed by rdfD1: 

_:dls a rdf:dirLangString .

3.2.1.1 Extended Directional Language String Interpretation

An extended directional language string interpretation is an RDFS interpretation I which also satisfies the following semantic condition, and all the triples in the subsequent table of axiomatic triples.

Directional language string semantic condition.

If IL(E) is in ICEXT(I(rdf:dirLangString)) and IL(E) = <v, l, d> where
v is the lexical form, l is the language tag in lower-case, and d is the direction of E,
then:

  • <IL(E), IL(v)> is in IEXT(I(rdf:value)),
  • <IL(E), IL(l)> is in IEXT(I(rdf:language)), and
  • <IL(E), IL(d)> is in IEXT(I(rdf:direction)).

and:

  • IL(v) is in ICEXT(I(xsd:string)),
  • IL(l) is in ICEXT(I(xsd:language)), and
  • IL(d) is in ICEXT(I(rdf:langDir)).
Axiomatic triples of directional language string forms. 
rdf:langDir rdf:type rdfs:Datatype .

rdf:language rdf:type rdf:Property .
rdf:language rdfs:domain [owl:unionOf (rdf:langString rdf:dirLangString)] .
rdf:language rdfs:range xsd:language .

rdf:direction rdf:type rdf:Property .
rdf:direction rdfs:domain rdf:dirLangString .
rdf:direction rdfs:range rdf:langDir .

3.2.2 Language Strings

The language string pattern: 

SSS PPP "..."@und .

can be described as: 

SSS PPP _:dls .
_:dls a rdf:langString .
_:dls rdf:value "..." .  # Use rdf:lexicalForm?
_:dls rdf:language "und"^^xsd:language .
3.2.2.1 Extended Language String Interpretation

The definition of a semantic condition for rdf:langString is similar to the semantic condition of rdf:dirLangString, but without the rdf:direction part:

Language string semantic condition.

If IL(E) is in ICEXT(I(rdf:langString)) and IL(E) = <v, l> where
v is the lexical form, and l is the language tag in lower-case of E,
then:

  • <IL(E), IL(v)> is in IEXT(I(rdf:value)), and
  • <IL(E), IL(l)> is in IEXT(I(rdf:language)).

and:

  • IL(v) is in ICEXT(I(xsd:string)), and
  • IL(l) is in ICEXT(I(xsd:language)).

3.2.3 Other Datatypes

In principle, any datatyped literals encode resources with intrinsic meaning, and that this meaning can be formally specified.

It is therefore generally possible to define semantic extensions for values of any formally defined datatype, entailing relationships between literal values and their constituent parts.

For example, a literal xsd:date value: 

SSS PPP "YYYY-MM-DD"^^xsd:date .

implies a resource with one year, one month and one day (here expressed using the Time Ontology in OWL): 

SSS PPP _:dt a xsd:date .
_:dt time:year "YYYY"^^xsd:gYear .
_:dt time:month "--MM"^^xsd:gMonth .
_:dt time:day "---DD"^^xsd:gDay .

Defining semantic conditions for such structures, and the formal properties thereof, is beyond the scope of this document (and outside the purview of RDF for any datatype defined by other specifications). A possible SPARQL-based expansion form is proposed in Appendix A.4 Date Time Form Expansion.

4. OWL-based Interoperability Patterns

Using the extended interpretations defined above results in a graph which can be further reasoned over through the semantic extension of OWL. Axioms can be defined to describe restriction subclasses of rdfs:Proposition based on, for example, a particular predicate, and to define sub-property chain axioms of relationships that hold between entities in more complex relationships. This allows for annotations of direct, more abstract statements to entail underlying, more granular forms of association classes.

Note

The patterns below utilize owl:hasSelf to define a rolification of a class, meaning a restriction pattern for the relationship from an individual to itself, which is equivalent to a specified class. This restriction implies that an instance of such a class has this rolified relationship to itself (and vice versa). That enables the use of the rolification property in sub-property chain axioms, functioning as a self-relation chain component matching the type of the resource.

4.1 Interoperability Patterns Using Reified Propositions

The following patterns rely on OWL 2 and extended interpretation of propositions to ensure that simplified descriptions leveraging reifying triples and/or triple annotations can be interoperably interpreted as classical, more complex forms of descriptions.

4.1.1 Classic RDF Reification as Reifiers

Given these axioms:

Example 1: classic statement model based on the propositions form 
rdf:subject
    owl:propertyChainAxiom (_:a_Statement rdf:reifies rdf:propositionFormSubject) .
rdf:predicate
    owl:propertyChainAxiom (_:a_Statement rdf:reifies rdf:propositionFormPredicate) .
rdf:object
    owl:propertyChainAxiom (_:a_Statement rdf:reifies rdf:propositionFormObject) .

[ owl:onProperty _:a_Statement ; owl:hasSelf true ]
    owl:equivalentClass rdf:Statement .

this graph:

Example 2: a reified statement 
PREFIX dct: <http://purl.org/dc/terms/>

<< <Alice> dct:creator <SomeComputer> >> a rdf:Statement ;
    dct:date "2014-12-15"^^xsd:date ;
    dct:source <SomeDatabase> .

entails this graph:

Example 3: a classic reified statement 
PREFIX dct: <http://purl.org/dc/terms/>

<< <Alice> dct:creator <SomeComputer> >> a rdf:Statement ;
    rdf:subject <Alice> ;
    rdf:predicate dct:creator ;
    rdf:object <SomeComputer> .
    dct:date "2014-12-15"^^xsd:date ;
    dct:source <SomeDatabase> .

4.1.2 PROV-O Qualifications as Reifiers

In the PROV Ontology ([PROV-O]) several association classes have been defined as qualified counterparts for a number of simple relationships. In this section, two examples are examined (borrowed from examples 8 and 9 of [PROV-O]). The following prefix is assumed in the subsequent examples: 

PREFIX prov: <http://www.w3.org/ns/prov#>

These two simple relationships:

Example 4: simple PROV-O relationships 
<illustration_activity> a prov:Activity;
  prov:used <aggregated_by_regions> ;
  prov:wasAssociatedWith <derek> .

have the following companion qualified forms:

Example 5: qualified PROV-O relationships 
<illustration_activity> a prov:Activity;
  prov:qualifiedUsage [ a prov:Usage;
      prov:entity <aggregated_by_regions>;
      prov:atTime "2011-07-14T03:03:03Z"^^xsd:dateTime ] ;
  prov:qualifiedAssociation [ a prov:Association;
      prov:agent   <derek>;
      prov:hadRole <illustrationist>;
      prov:hadPlan <tutorial_blog>
    ] .

Using OWL, there is a way to entail the simple relationships from the qualified forms:

Example 6: OWL axioms for entailing simple relationships 
prov:used owl:propertyChainAxiom (prov:qualifiedUsage prov:entity) .
prov:wasAssociatedWith owl:propertyChainAxiom (prov:qualifiedAssociation prov:agent) .

With the extended interpretation of propositions, it is possible to conversely entail the qualified forms from annotated versions of the simple relationships:

Example 7: annotated simple PROV-O relationships 
<illustration_activity> a prov:Activity;
  prov:used <aggregated_by_regions> {| a prov:Usage ;
      prov:atTime "2011-07-14T03:03:03Z"^^xsd:dateTime |} ;
  prov:wasAssociatedWith <derek> {| a prov:Association ;
      prov:hadRole <illustrationist>;
      prov:hadPlan <tutorial_blog> |} .

This can be done by extending the ontology itself, by defining property chains between reifiers of type prov:Usage and the corresponding subject and object of a proposition with the prov:used predicate:

Example 8: OWL axioms for entailing qualified usage relationship 
prov:qualifiedUsage
    owl:propertyChainAxiom (
            [owl:inverseOf rdf:propositionFormSubject]
            [owl:inverseOf _:reified_used_ppn]
        ) .

prov:entity owl:propertyChainAxiom (_:reified_used_ppn rdf:propositionFormObject) .

_:reified_used_ppn owl:propertyChainAxiom (_:a_Usage rdf:reifies _:used_ppn) .

[ owl:onProperty _:a_Usage ; owl:hasSelf true ]
    owl:equivalentClass prov:Usage .

[ owl:onProperty _:used_ppn ; owl:hasSelf true ]
    owl:equivalentClass
        [ owl:onProperty rdf:propositionFormPredicate ;
          owl:hasValue prov:used ] .

and similarly for reifiers of type prov:Association of triples with the prov:wasAssociatedWith predicate:

Example 9: OWL axioms for entailing qualified association relationship 
prov:qualifiedAssociation
    owl:propertyChainAxiom (
            [owl:inverseOf rdf:propositionFormSubject]
            [owl:inverseOf _:reified_assocWith_ppn]
        ) .

prov:agent owl:propertyChainAxiom (_:reified_assocWith_ppn rdf:propositionFormObject) .

_:reified_assocWith_ppn
    owl:propertyChainAxiom (_:a_Association rdf:reifies _:associated_ppn) .

[ owl:onProperty _:a_Association ; owl:hasSelf true ]
    owl:equivalentClass prov:Association .

[ owl:onProperty _:associated_ppn ; owl:hasSelf true ]
    owl:equivalentClass
        [ owl:onProperty rdf:propositionFormPredicate ;
        owl:hasValue prov:wasAssociatedWith ] .

4.1.3 Events as Proposition-reifying Truthmakers

The possibility of referring to the constituent properties of a proposition also makes it possible to define OWL-based axioms through which a proposition is entailed to hold. (Meaning that, if entailments are materialized, a triple denoting such a proposition is asserted.)

The following exemplifies that, by entailing a certain kind of proposition to be the object of an rdf:type relationship to its subject, and further that this kind of proposition is a restriction class on its subject, such that this subject has the object of the proposition as the value of the predicate of the proposition.

The class of events is further defined to reify propositions of this particular kind; specifically those being objects of an rdf:reifies relationship from event instances. This can be thought of as events being truthmakers of these (truthbearing) propositions.

An event reifying a proposition:

Example 10: simple PROV-O relationships 
PREFIX ex: <https://example.net/ns/>
BASE <http://example.net/>

<< <Theseus> ex:used <Ship1> >> ex:atTime -101861280000000 .

makes the reified proposition hold:

Example 11: simple PROV-O relationships 
PREFIX ex: <https://example.net/ns/>
BASE <http://example.net/>

<Theseus> ex:used <Ship1> {| a ex:Event ; ex:atTime -101861280000000 |} .

given the following axioms:

Example 12: simple PROV-O relationships 
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX owl: <http://www.w3.org/2002/07/owl#>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
PREFIX ex: <https://example.net/ns/>

##
# Anything with an at time property is an Event:
ex:atTime rdfs:domain ex:Event .

##
# A rolification of the class of propositions which are reified by events:
[ owl:onProperty _:a_ppn_reifiedby_a_Event ; owl:hasSelf true ]
  owl:equivalentClass
    [ owl:onProperty [owl:inverseOf rdf:reifies] ; owl:someValuesFrom ex:Event ] .

##
# The restriction class of propositions that reify events.

# The subject of an event-reified proposition has the proposition as its type.
[ owl:inverseOf rdf:type ]
  owl:propertyChainAxiom
    ( _:a_ppn_reifiedby_a_Event rdf:propositionFormSubject ) .

# The predicate of an event-reified proposition is the property of the
# restriction class:
owl:onProperty
  owl:propertyChainAxiom
    ( _:a_ppn_reifiedby_a_Event rdf:propositionFormPredicate ) .

# The object of an event-reified proposition is the value of the restriction
# class:
owl:hasValue
  owl:propertyChainAxiom
    ( _:a_ppn_reifiedby_a_Event rdf:propositionFormObject ) .

The above pattern declares propositions that are reified by events as restriction types on their subject, such that they hold in the interpretation.

5. Appendices

A. Structural Expansions using SPARQL

The following subsections define SPARQL queries for constructing descriptions of the expanded formal structures. They have to be repeatedly run against the union of the source graph and each result graph, until the triple count is equal to the triple count of the previous iteration.

Note

The construct patterns below use owl:sameAs to encode equivalence of the denoting terms with the blank nodes used as subjects for the encoded descriptions. Other semantic extensions that do not recognize OWL 2 semantics may require other forms for encoding the entailments, such as symmetric RDF.

A.1 Proposition Form Expansion

Example 13: SPARQL-based propositions form expansion 
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>

CONSTRUCT {
  ?s ?rel ?ppn .
  ?ppn a rdf:PropositionForm ;
    owl:sameAs <<( ?ppn_s ?ppn_p ?ppn_o )>> ;
    rdf:propositionFormSubject ?ppn_s ;
    rdf:propositionFormPredicate ?ppn_p ;
    rdf:propositionFormObject ?ppn_o .
} WHERE {
  ?s ?rel <<( ?ppn_s ?ppn_p ?ppn_o )>> .
  OPTIONAL {
    ?existing_ppn a rdf:PropositionForm ;
      rdf:propositionFormSubject ?ppn_s ;
      rdf:propositionFormPredicate ?ppn_p ;
      rdf:propositionFormObject ?ppn_o .
  }
  BIND(COALESCE(?existing_ppn, BNODE()) as ?ppn)
}

A.2 Directional Language String Form Expansion

Example 14: SPARQL-based directional language string expansion 
# Standard RDF, relying on owl:sameAs. For Symmetric RDF: ?strlangdir a ...
CONSTRUCT {
  ?s ?p [ a rdf:dirLangString ;
          owl:sameAs ?strlangdir ;
          rdf:value ?lexical ;
          rdf:language ?lang ;
          rdf:direction ?langdir ] .
} WHERE {
  ?s ?p ?strlangdir .
  FILTER(hasLANG(?strlangdir) && hasLANGDIR(?strlangdir))
  BIND(STR(?strlangdir) as ?lexical)
  BIND(LANG(?strlangdir) as ?lang)
  BIND(LANGDIR(?strlangdir) as ?langdir)
}

A.3 Language String Form Expansion

Example 15: SPARQL-based language string expansion 
CONSTRUCT {
  ?s ?p [ a rdf:langString ;
          owl:sameAs ?strlang ;
          rdf:value ?lexical ;
          rdf:language ?lang ] .
} WHERE {
  ?s ?p ?strlang .
  FILTER hasLANG(?strlang)
  BIND(STR(?strlang) as ?lexical)
  BIND(LANG(?strlang) as ?lang)
}

A.4 Date Time Form Expansion

This is only a suggested expansion form; based on properties defined by [OWL-TIME], beyond the purview of RDF.

Example 16: SPARQL-based datetime expansion 
PREFIX time: <http://www.w3.org/2006/time#>

CONSTRUCT {

  ?s ?p ?matchedform .

  ?timeurn a ?type ;
    owl:sameAs ?time ;
    time:inXSDDate ?datevalue ;
    time:hour ?hours ;
    time:minute ?minutes ;
    time:second ?seconds ;
    time:timeZone ?tzNode .

  ?dateurn a xsd:date ;
    owl:sameAs ?datevalue ;
    time:year ?year ;
    time:month ?month ;
    time:day ?day .

  ?tzNode a time:TimeZone ;
    rdf:value ?tz, ?timezone .

} WHERE {

  {
    ?s ?p ?time .
    BIND(DATATYPE(?time) AS ?type)
    FILTER(?type IN (xsd:dateTimeStamp, xsd:dateTime))
    BIND(?time AS ?hasdate)
    BIND(IRI(CONCAT('urn:tdb:2025:urn:sha:',
                    SHA256(CONCAT(STR(?time),
                    '^^', STR(?type))))) AS ?timeurn)
  } UNION {
    ?s ?p ?date .
    BIND(DATATYPE(?date) AS ?type)
    FILTER(?type = xsd:date)
    BIND(?date AS ?hasdate)
  }

  BIND(STR(YEAR(?hasdate)) AS ?ystr)
  BIND(STRDT(?ystr, xsd:gYear) AS ?year)

  BIND(CONCAT(IF(MONTH(?hasdate) < 10, "0", ""), STR(MONTH(?hasdate))) AS ?mstr)
  BIND(STRDT(CONCAT("--", ?mstr), xsd:gMonth) AS ?month)

  BIND(CONCAT(IF(DAY(?hasdate) < 10, "0", ""), STR(DAY(?hasdate))) AS ?dstr)
  BIND(STRDT(CONCAT("---", ?dstr), xsd:gDay) AS ?day)

  BIND(COALESCE(?date, STRDT(CONCAT(?ystr, '-', ?mstr, '-', ?dstr),
                                    xsd:date)) AS ?datevalue)

  BIND(IRI(CONCAT('urn:tdb:2025:urn:sha:',
                  SHA256(CONCAT(?ystr, '-', ?mstr, '-', ?dstr,
                  '^^', STR(xsd:date))))) AS ?dateurn)

  BIND(STRDT(STR(HOURS(?time)), xsd:nonNegativeInteger) AS ?hours)
  BIND(STRDT(STR(MINUTES(?time)), xsd:nonNegativeInteger) AS ?minutes)
  BIND(SECONDS(?time) AS ?seconds)

  BIND(TZ(?time) AS ?tz)
  BIND(TIMEZONE(?time) AS ?timezone)

  BIND(IRI(CONCAT('urn:tdb:2025:urn:sha:',
                  SHA256(CONCAT(STR(?timezone),
                  '^^', STR(time:TimeZone))))) AS ?tzNode)

  BIND(IF(BOUND(?time), ?timeurn, ?dateurn) AS ?matchedform)

}

B. Relationship to Basic-encoding

The semantic conditions specified here entail structures that are structurally based on, but formally different from—or complementary to—the interoperability patterns between RDF 1.2 Full, RDF 1.2 Basic, and RDF 1.1 defined by the algorithms of basic encoding and decoding.

The inherent structures of propositions and datatype properties are possible to query over, either using SPARQL 1.2, or using SPARQL 1.1 along with basic encoding under simple entailment. But it is not possible to reason over them using RDF semantics alone. This prevents the definition of OWL-based axioms relating to this inherent structure. By defining a set of semantic extensions, this document bridges that gap.

These two methods are complementary ways of working with RDF 1.2 data. Basic-encoding is for RDF 1.1 or 1.2 Basic systems that lack support for triple terms. Extended interpretations of formal structures is for systems leveraging semantic extensions based on RDF 1.2, such as OWL 2, in order to be able to reason over the intrinsic properties of propositions and literal values.

B.1 Conditions for Basic-encoding

Basic-encoding and subsequent decoding is only safe under simple entailment.

Since literal values and propositions are resources, RDF entailment implies they have an rdf:type (of rdfs:Literal and rdfs:Proposition, respectively). Any encoding and decoding must take place on a graph where no entailments have been manifested.

While the RDF abstract syntax prevents this by disallowing the special encoding terms denoting these kinds of resources as subjects, the same is not possible if facts about them are encoded as triples with IRIs or blank nodes as subject identifiers. To avoid the direct logical description of these resources, basic encoding uses underlying base types for describing propositions and directional language strings, simply encoding their forms.

Note

Note that due to the rules of basic-encoding, it is impossible to basic-decode a graph representing the formal entailments about propositions (since their type is rdfs:Proposition). This is by design, to avoid basic-encoding data using anything beyond simple entailment. Entailed facts should not be put back into an RDF Source. They are simply true, following from the known axioms and conditions of a given entailment regime.

B.2 Upgrading to Structural Interpretations

Since applications may start with a dependency on deployed structures, and rely on basic-encoding, having a formal way of instead entailing these facts ensures a seamless upgrade path to RDF 1.2 Full, without requiring redefined domain logic or queries rewritten for new structures.

Note

Similarly, a basic-encoded graph will work mostly the same when reasoned over using OWL. One crucial difference is that there will be no triple terms in a basic-encoded graph, meaning that there is no owl:sameAs relationship to such terms from the encoded propositions. This is not significant for the meaning, but for implementations also relying on the graph structure itself.

B.3 Basic-encoding using SPARQL

The following subsections define SPARQL updates for basic-encoding and -decoding. They have to be repeated until the triple count is stable (i.e. iterate while the triple count increases (for encoding) or decreases (for decoding); and halting when the triple count is equal to the one of the previous iteration).

Note

Since datatyped literals and language-tagged strings have been part of RDF since version 1.0, there is no need for basic encoding such structures.

B.3.1 Basic-encoding of Propositions

Example 17: SPARQL-based basic-encoding of propositions 
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>

DELETE {
  ?s ?rel <<( ?ppn_s ?ppn_p ?ppn_o )>> .
}
INSERT {
  ?s ?rel ?ppn .
  ?ppn a rdf:PropositionForm ;
    rdf:propositionFormSubject ?ppn_s ;
    rdf:propositionFormPredicate ?ppn_p ;
    rdf:propositionFormObject ?ppn_o .
}
WHERE {
  ?s ?rel <<( ?ppn_s ?ppn_p ?ppn_o )>> .
  OPTIONAL {
    ?existing_ppn a rdf:PropositionForm ;
      rdf:propositionFormSubject ?ppn_s ;
      rdf:propositionFormPredicate ?ppn_p ;
      rdf:propositionFormObject ?ppn_o .
  }
  BIND(COALESCE(?existing_ppn, BNODE()) as ?ppn)
}

B.3.2 Basic-encoding of Directional Language Strings

Example 19: SPARQL-based basic-encoding of dir-lang-strings 
DELETE {
  ?s ?p ?strlangdir .
}
INSERT {
  ?s ?p ?litnode .
  ?litnode a rdf:CompoundLiteral ;
    rdf:value ?lexical ;
    rdf:language ?lang ;
    rdf:direction ?langdir .
}
WHERE {
  ?s ?p ?strlangdir .
  FILTER(hasLANG(?strlangdir) && hasLANGDIR(?strlangdir))
  BIND(STR(?strlangdir) as ?lexical)
  BIND(LANG(?strlangdir) as ?lang)
  BIND(LANGDIR(?strlangdir) as ?langdir)
  OPTIONAL {
    ?existing_litnode a rdf:CompoundLiteral ;
      rdf:value ?lexical ;
      rdf:language ?lang ;
      rdf:direction ?langdir .
  }
  BIND(COALESCE(?existing_litnode, BNODE()) as ?litnode)
}
Example 20: SPARQL-based basic-decoding of dir-lang-strings 
DELETE {
  ?s ?p ?litnode .
  ?litnode a rdf:CompoundLiteral ;
    rdf:language ?lang ;
    rdf:direction ?langdir .
}
INSERT {
  ?s ?p ?strlangdir .
}
WHERE {
  ?s ?p ?litnode .
  ?litnode a rdf:CompoundLiteral ;
    rdf:language ?lang ;
    rdf:direction ?langdir .

  BIND(STRLANGDIR(STR(?lexical), ?lang, ?langdir) as ?strlangdir)
}
